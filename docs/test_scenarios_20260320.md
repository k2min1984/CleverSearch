> **(대상 일정: 39, 50, 51, 52)**

2.  **쓰기 권한 확인**: `사전관리`, `스케줄러`, `초기화` 등 설정 변경이나 실행 관련 버튼이 비활성화되어 있거나 클릭해도 동작하지 않는지 확인합니다.
3.  **로그아웃 및 `admin` 로그인**: 로그아웃 후 `admin` 계정으로 다시 로그인하여 모든 기능이 활성화되는지 확인합니다.

### 1-2. 파일 색인 및 중복 방지

> **(대상 일정: 8, 11, 13, 15, 17, 18, 20, 63)**

1.  **파일 업로드**: `파일 업로드` 탭에서 준비된 테스트 파일 5개를 한 번에 업로드합니다.
    - **기대**: 5개 파일 모두 `success` 상태로 표시됩니다.
2.  **색인 확인**: `문서 목록` 탭에서 목록이 5건으로 표시되는지 확인합니다. 각 문서의 본문 미리보기를 통해 텍스트가 깨짐 없이 정상적으로 추출되었는지 확인합니다.
3.  **중복 업로드 (동일 파일)**: `파일 업로드` 탭에서 방금 올린 파일 중 하나(`연구개발계획서_작성중.docx`)를 다시 업로드합니다.
    - **기대**: `skipped` 상태로 표시되며, `문서 목록`의 총 건수는 변함이 없습니다.
4.  **중복 업로드 (동일 내용, 다른 이름)**: `연구개발계획서_작성중.docx` 파일의 복사본을 만들어 이름을 바꾸고 업로드합니다.
    - **기대**: 내용 해시(Hash) 기반 중복 감지로 `skipped` 처리됩니다.

### 1-3. 검색 품질 관리

> **(대상 일정: 31, 37, 38)**

1.  **가중치 변경**: `검색 품질` 탭에서 '제목 구문' 가중치를 `20`에서 `30`으로 변경하고 **[저장]** 버튼을 클릭합니다.
    - **기대**: "저장 완료" 메시지가 표시됩니다.
2.  **변경 확인**: 사용자 페이지에서 '제목 구문'과 관련된 검색(예: `연구개발계획서`)을 수행하여 순위 변동을 확인합니다. (필수는 아님)
3.  **초기값 복원**: `검색 품질` 탭에서 **[초기값 복원]** 버튼을 클릭합니다.
    - **기대**: 모든 가중치가 기본값으로 복원됩니다.

### 1-4. 로그 및 통계 연동

> **(대상 일정: 45, 62)**

1.  **로그 생성**: 사용자 페이지에서 **시나리오 2**의 검색들을 모두 수행하여 검색 로그를 생성합니다.
2.  **로그 확인**: `검색 로그` 탭에서 방금 수행한 검색 기록(시간, 키워드, 결과 수)이 모두 표시되는지 확인합니다.
3.  **실패 로그 확인**: `실패 검색어` 탭에서 결과가 0건이었던 검색어(예: `ZXCVBNM...`)만 집계되어 표시되는지 확인합니다.
4.  **인기 검색어 확인**: `인기검색` 탭에서 검색 빈도가 높은 순서대로 키워드가 정렬되어 표시되는지 확인합니다.

## 부록 A. 파일 업로드/검색어 저장 방식 쉬운 설명 (공유용)

이 섹션은 "어떤 방식"을 쓰는지와 "왜 그렇게 했는지"를 비개발자도 설명할 수 있게 정리한 내용입니다.

### A-1. 파일 업로드 저장 방식

#### 1) 한 줄 요약

- **파일은 OpenSearch에 검색용으로 색인하고, 앱 DB에는 관리용 메타정보를 함께 저장하는 이중 구조**를 사용합니다.

#### 2) 실제 처리 순서

1. 사용자가 파일 업로드
2. 서버가 파일명/확장자/용량을 먼저 검사 (허용 확장자, 최대 용량)
3. 파일 바이트로 SHA-256 해시를 계산
4. 해시가 같으면 **중복으로 판단하고 건너뜀(skipped)**
5. 중복이 아니면 텍스트 추출(OCR 포함) 후 OpenSearch에 색인
6. 동시에 앱 DB(`indexed_documents`)에 문서 메타/해시를 저장

#### 3) 왜 이 방식을 쓰는가

- **검색 성능**: OpenSearch는 검색/필터/랭킹에 최적화되어 있어 조회가 빠릅니다.
- **운영 안정성**: 앱 DB는 문서 목록, 중복 이력, 관리 화면 집계에 안정적입니다.
- **중복 방지 정확도**: 파일명 기준이 아니라 **내용(해시) 기준**이라 이름만 바꿔도 중복을 잡습니다.
- **비용 절감**: 중복 색인을 막아 저장공간과 색인 시간을 줄입니다.

#### 4) 다른 사람에게 이렇게 설명하면 됩니다

- "우리는 파일을 두 군데에 저장하는 게 아니라, 역할을 나눠서 관리해요. 검색은 OpenSearch, 운영 이력은 DB. 그래서 빠르고 안전해요."

### A-2. 검색어 저장 방식

#### 1) 한 줄 요약

- 검색이 실행되면 **검색 로그(`search_logs`)**와 **최근 검색어(`recent_searches`)**를 앱 DB에 저장합니다.

#### 2) 실제 저장 규칙

1. 사용자가 검색 실행
2. 결과 건수(`total_hits`)를 기준으로 실패 여부 판정 (0건이면 실패)
3. `search_logs`에 사용자ID, 검색어, 결과건수, 실패여부, 시각 저장
4. `recent_searches`에는 사용자별 최근 검색어 저장
5. 같은 검색어를 다시 검색하면 새 행을 무한 추가하지 않고 **시간만 갱신**
6. 최근 검색어는 사용자별 보관 개수 제한(기본 20개)으로 오래된 항목 자동 정리

#### 3) 왜 이 방식을 쓰는가

- **분석 가능**: 실패 검색어, 인기 검색어, 추천 검색어를 로그에서 만들 수 있습니다.
- **UX 개선**: 최근 검색어/인기 검색어/추천 검색어 기능이 바로 동작합니다.
- **데이터 폭증 방지**: 최근 검색어를 중복 누적하지 않고 최신화 + 개수 제한으로 관리합니다.
- **운영 추적성**: 누가 어떤 검색을 했는지, 결과가 몇 건인지 근거를 남길 수 있습니다.

#### 4) 다른 사람에게 이렇게 설명하면 됩니다

- "검색어는 단순 기록이 아니라 품질 개선 데이터예요. 로그가 있어야 실패 키워드 개선, 인기 키워드 노출, 개인화 추천이 가능합니다."

### A-3. 자주 받는 질문(FAQ)

#### Q1. 파일을 왜 DB와 OpenSearch 둘 다 쓰나요?

- A. 한 시스템에 몰아넣으면 한쪽 기능이 약해집니다. 검색은 OpenSearch가 강하고, 운영/관리/통계는 관계형 DB가 강합니다.

#### Q2. 중복 체크를 파일명으로만 하면 안 되나요?

- A. 파일명은 쉽게 바꿀 수 있어 정확하지 않습니다. 내용 해시(SHA-256) 기준이 가장 신뢰도가 높습니다.

#### Q3. 검색어 저장이 개인정보 이슈는 없나요?

- A. 현재는 `user_id`와 검색어를 운영 목적(품질 개선/통계)으로 저장합니다. 운영 정책에 따라 마스킹/보관기간/삭제정책을 추가하면 더 안전합니다.

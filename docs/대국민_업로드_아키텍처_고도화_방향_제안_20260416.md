# 대국민 업로드 고도화 방향 요약 (2026-04-16)

## 1. 한 줄 결론

현재 방식(요청 중 실시간 처리)에서, 접수와 처리를 분리한 비동기 방식으로 전환해야 한다.

## 2. 지금 상태 (현재 구조)

- 업로드 요청이 들어오면 API 서버가 파일 검증, 텍스트 추출, 색인까지 한 번에 처리한다.
- 업로드 중 화면 잠금이 발생해 다른 작업이 어렵다.
- 대용량 파일 전송 중 끊기면 처음부터 다시 보내야 한다.

근거:

- [static/admin/upload.html](../static/admin/upload.html)
- [app/api/v1/file.py](../app/api/v1/file.py)
- [app/services/indexing_service.py](../app/services/indexing_service.py)

## 3. 왜 바꿔야 하나

- 대량 업로드 시 응답 지연과 서버 병목이 생기기 쉽다.
- 사용자 체감이 불편하다(대기, 재시작, 화면 잠금).
- 운영 관점에서 작업 상태 추적/재처리가 어렵다.

## 4. 목표 구조

- 업로드는 "접수"만 빠르게 완료하고 작업번호를 즉시 반환한다.
- 실제 처리는 백그라운드 워커가 진행한다.
- 관리자는 상단 팝업에서 전체/개별 진행률과 남은시간을 본다.
- 레이어 팝업에서 파일별 취소와 일괄 취소를 지원한다.
- 로그아웃 시 사용자에게 선택권을 부여하되, 기본값은 계속 진행이다.
- 재로그인 시 미완료 업로드가 있으면 상단 최소화 팝업을 자동 노출한다.

## 5. 청크 업로드가 필요한 이유

청크 업로드는 큰 파일을 작은 조각으로 나눠 보내는 방식이다.

예시:

- 기존: 1GB 파일을 한 번에 전송 -> 99%에서 끊기면 처음부터 다시 전송
- 청크: 1GB를 10MB씩 전송 -> 중간에 끊기면 마지막 성공 지점부터 재개

효과:

- 업로드 실패 재시작 부담 감소
- 불안정한 네트워크에서 완료율 상승
- 재전송 데이터 감소로 시간/비용 절감
- 초대형 단건 요청으로 인한 서버 부담 완화

## 6. 로그아웃 정책 [결정: 사용자 선택(User Choice)]

### 기본 방침

로그아웃 시 업로드 중인 파일이 있으면 팝업으로 사용자에게 선택권을 부여한다.

```
⚠️ 업로드 중인 파일이 있습니다.

[① 백그라운드에서 계속 진행] (기본값)
   → 로그아웃 후에도 서버에서 계속 처리됩니다.

[② 업로드를 취소하고 로그아웃]
   → 진행 중인 모든 업로드가 취소됩니다.
```

### 3가지 안 비교 및 결정 근거

| 정책            | 장점                        | 단점                               | 대국민 서비스 적합도 |
| --------------- | --------------------------- | ---------------------------------- | -------------------- |
| 즉시 취소       | 보안상 깔끔                 | 대용량 업로드 중 실수 시 민원 폭주 | ❌ 부적합            |
| 계속 진행       | 사용자 편의 최대            | 공공PC에서 파일 잔류 보안 리스크   | ⚠️ 단독 채택 위험    |
| **사용자 선택** | 상황별 최적 대응, 양쪽 만족 | UI/서버 로직 복잡도 증가           | **✅ 채택**          |

### 물리적 상태 구분 (온프레미스 전제)

**상황 A: 브라우저 → 서버 전송 중일 때 로그아웃/창 닫기**

- 브라우저가 닫히면 네트워크 연결이 끊기므로 서버가 파일을 당겨올 방법이 없다.
- 이때는 `일시 중지(PAUSED)` 상태가 된다.
- 재로그인 시 최소화 팝업에 "n건의 일시 중지된 업로드가 있습니다. [이어서 전송]" 버튼을 노출한다.
- 청크 이어올리기(Resume)로 마지막 성공 청크부터 재개한다.

**상황 B: 서버 전송 완료 후, 워커가 색인/추출 중일 때**

- 서버가 파일을 이미 갖고 있으므로 계속 진행이 완벽히 가능하다.
- 팝업에 "업로드 완료, 서버에서 분석 중입니다. 창을 닫으셔도 좋습니다."를 표시한다.

### 필수 조건

- 사용자별 작업 접근 제어
- 재로그인 시 진행 상태 자동 복구 (최소화 팝업)
- 작업 이력(시작/완료/실패/일시중지) 기록
- 재로그인 직후 미완료 건은 기본 "최소화 팝업"으로 노출, 사용자가 확장해 상세 확인

### 일시 중지(PAUSED) 전환 조건

다음 상황에서 해당 업로드 작업은 자동으로 PAUSED 상태로 전환된다.

1. 브라우저 탭/창을 강제 종료한 경우 (네트워크 연결 끊김 감지)
2. 네트워크 장애로 5분 이상 청크 전송이 없는 경우
3. 로그아웃 팝업에서 "취소하고 로그아웃"을 선택하지 않았으나, 전송 중인 파일이 남아 있는 경우

PAUSED 상태의 파일은 재로그인 시 "이어서 전송" 버튼으로 재개 가능하며, 24시간이 지나도 재개되지 않으면 GC 정책(6-1-8)에 따라 삭제된다.

### 취소 정책

- 레이어 팝업에 파일별 "취소" 버튼과 "전체 취소" 버튼을 둔다.
- 취소 대상은 대기/업로드중/처리중 파일이며, 취소 즉시 사용자 화면에 상태를 반영한다.
- 취소된 건은 초기화 메뉴와 동일한 기준으로 정리한다.
- 즉, OpenSearch 색인 데이터와 DB 내 해당 인덱스/메타데이터를 함께 삭제한다.
- 삭제 실패 시 재시도 큐로 넘겨 최종 정합성을 맞추고, 이벤트 로그를 남긴다.

## 6-1. 핵심 운영 정책 [결정: 채택]

표시 기준: [결정: 채택]은 현재 기준안으로 확정해 구현한다는 의미

### 1) 취소/정리 정책 [결정: 채택]

- 레이어 팝업에서 파일별 취소, 일괄 취소를 모두 지원한다.
- 취소 완료 기준은 "사용자 화면 반영"이 아니라 "OpenSearch+DB 정리 완료"로 본다.
- 부분 실패가 나면 보상 재시도 큐에서 처리하고, 완료 전까지 상태를 `CANCELING`으로 유지한다.

### 2) 상태 표시 정책 [결정: 채택]

- 상단 팝업은 `확장/최소화/복원` 3가지 상태를 유지한다.
- 재로그인 시 미완료 건이 있으면 기본은 최소화 상태로 자동 노출한다.
- 팝업에는 전체 진행률, 파일별 진행률, 실패/취소 건수를 함께 표시한다.

### 3) 접근 제어 정책 [결정: 채택]

- 업로드 작업 조회/취소는 작업 생성 사용자와 관리자만 가능하다.
- 사용자 A는 사용자 B의 작업 상세를 볼 수 없고 취소도 불가하다.
- 모든 조회/취소 API는 인증 토큰과 사용자 권한 검증을 통과해야 한다.

### 4) 정합성 정책 [결정: 채택]

- 색인 생성/삭제는 DB 상태와 OpenSearch 상태를 이벤트 단위로 맞춘다.
- 한쪽만 반영된 경우 `INCONSISTENT`로 마킹하고 자동 복구 작업을 실행한다.
- 복구가 일정 시간 내 실패하면 운영 알림을 발생시킨다.

### 5) 재시도/실패 처리 정책 [결정: 채택]

- 네트워크/일시 오류는 자동 재시도(지수 백오프) 후 DLQ로 이관한다.
- 영구 실패는 사용자 화면에 사유를 표시하고 재실행 버튼을 제공한다.
  ex> 경로 A: 일시적인 오류 (네트워크 끊김, 서버 과부하)
  시스템이 뒤에서 여러 번 재시도 -> 그래도 안 되면 DLQ에 저장 (관리자가 나중에 처리) -> 동시에 사용자 화면에 "서버 지연으로 실패했습니다" + [재실행] 버튼 노출
  ex> 경로 B: 영구 실패 (파일 손상, 허용되지 않는 파일)
  시스템이 재시도해도 해결 불가한 문제이므로 자동 재시도/DLQ 과정을 생략한다.
  즉시 포기하고, 사용자 화면에 "파일이 손상되어 읽을 수 없습니다" 등 정확한 사유 + [재실행] 버튼을 노출한다. 사용자가 정상 파일로 교체해서 다시 올릴 수 있게 한다.
- 실패 이력은 `job_event`에 남겨 감사 추적이 가능해야 한다.

### 6) 감사/운영 정책 [결정: 채택]

- 주요 이벤트(접수, 시작, 완료, 실패, 취소, 삭제)는 모두 감사 로그에 기록한다.
- 운영자는 작업별 타임라인과 실패 원인을 조회할 수 있어야 한다.
- 일일 집계(성공률, 평균 처리시간, 취소율)를 운영 리포트로 남긴다.

### 7) 덮어쓰기(문서 최신화) 정책 [결정: 채택]

- 파일명(또는 해시)이 동일한 파일이 드롭될 경우, 레이어 팝업에서 충돌 경고 모달을 노출한다.
- 선택지: `[최신 버전으로 업데이트(덮어쓰기)]` / `[건너뛰기]`
- 검색 품질 보호: '새 이름으로 저장(Keep Both)' 기능은 통합 검색 결과의 중복(검색 공해)을 유발하므로 기본으로 제공하지 않는다.
- 덮어쓰기 시 기존 OpenSearch 문서를 삭제(또는 Update)하고 새 내용으로 교체한다.
- 다중 파일 업로드에서는 "이번 선택을 전체에 적용" 옵션을 제공한다.
- 명시적 교체가 필요한 경우 문서 목록에서 "파일 교체" 기능을 별도 지원한다.

### 8) 보안 및 인프라(온프레미스) 보호 정책 [결정: 채택]

- **미완료 파일 자동 정리(GC)**: 업로드 중단 후 24시간 경과한 청크 조각 및 임시 파일은 스케줄러(Celery Beat/Cron)로 매일 새벽 영구 삭제하여 스토리지 고갈을 방지한다.
- **워커 리소스 격리**: 악의적 대용량/압축 해제 문서(Zip Bomb)로 인한 CPU/메모리 고갈을 막기 위해, 워커의 텍스트 추출 작업에 엄격한 Timeout(파일당 3분)과 메모리 제한을 설정한다.
- **MIME 스푸핑 방어**: 확장자는 .pdf인데 실제 내용이 악성 실행 파일인 경우를 막기 위해, 첫 번째 청크(Header) 수신 시 python-magic 등으로 파일 시그니처를 즉시 검증한다. 조립 후 검사하면 늦다.
- **웹 접근성(WA) 준수**: 최소화된 업로드 팝업의 진행률 변화는 유의미한 상태(시작, 50%, 완료, 실패)에만 `aria-live="polite"`를 통해 스크린 리더에 음성 안내되도록 구현한다. 덮어쓰기 확인 모달은 Focus Trapping(초점 가두기)을 적용해 Tab 키가 모달 밖으로 넘어가지 않게 한다.

### 9) 재시도 멱등성 정책 [결정: 채택]

- 모든 업로드 접수 API는 클라이언트가 생성한 `idempotency_key`를 받는다.
- 같은 `idempotency_key`로 여러 번 요청해도 서버는 한 번만 처리하고 같은 응답을 반환한다.
- 네트워크 오류로 재전송된 요청이 중복 처리되는 것을 방지한다.

```json
POST /api/v1/upload-jobs
{
  "idempotency_key": "client_generated_uuid_12345",
  "files": [...]
}
```

## 7. 단계별 추진

### Phase 1 (1~2주)

- 상단 업로드 팝업(확장/최소화/복원) 적용
- 상태 폴링으로 진행률 표시
- 화면 잠금 최소화
- 로그아웃 시 경고 팝업만 구현 ("업로드 중입니다. 정말 나가시겠습니까?", 선택 분기 없음)

### Phase 2 (3~6주)

- 로그아웃 시 '전송 중 vs 처리 중' 분기 처리 + 사용자 선택 팝업 고도화
- `upload-jobs` 접수 API 신설 (멱등성 키 포함)
- 파일별 취소 API, 일괄 취소 API 신설
- Redis + Celery(RQ 대안 가능) 도입
- 워커 처리 단계화: 저장 -> 검사(악성코드 AV스캔 포함) -> 추출 -> 색인
- job/job_item/job_event 테이블 도입
- 청크 API 설계: init / upload-part / complete / abort
- 파일 정체성(SHA-256) 중복 감지 로직 구현
- 덮어쓰기 충돌 모달 + target_file_id 기반 교체 API 구현
- 미완료 청크 GC 스케줄러 구현 (24시간 경과 시 삭제)

### Phase 3 (6주+)

- 우선순위 큐, 자동 재시도, DLQ 운영
- 모니터링/알림 및 SLO 수립
- 분산 스토리지 연동(NAS/MinIO)
- 청크 이어올리기(Resume) 운영 안정화
- 저장소 생명주기(60일 보관/자동 삭제) 운영
- 임시 저장(Draft) 기능 (선택적)
- 스마트 중복 감지 (해시 기반, 파일명 무관)
- 업로드 히스토리 타임라인 뷰
- 파일 교체 시 텍스트 diff 비교 뷰어 (선택적)

## 8. 제한 정책(초기 권장값)

- 파일 1개 최대: 100MB
- 웹 1회 업로드 파일 수: 20개
- 웹 1회 총합: 500MB
- 비동기 잡 1회 접수 최대: 100개
- 워커 동시 처리: 3~5
- 청크 크기: 5~10MB (초기 8MB 권장)

## 8-1. 저장소 생명주기 정책 [결정: 채택]

- 업로드된 파일의 기본 보관 기간: **60일**
- 보관 기간 만료 시:
  - 7일 전 예고 알림 발송
  - 자동 삭제 (OpenSearch + DB + 물리 파일)
  - 삭제 이력은 1년간 보관
- 저장소 사용률 **80%** 초과 시 운영자 알림
- 저장소 사용률 **90%** 초과 시 신규 업로드 차단 + 사용자 안내
- 미완료 청크 임시 파일: 24시간 경과 후 GC 스케줄러로 자동 삭제

## 10. 파일 업로드 상세 설계 [결정: 채택]

### 10-1) 업로드 UI 정책

- 레이어 팝업은 파일명, 확장자, 크기, 상태(대기/업로드중/처리중/완료/실패/취소/일시중지), 진행률, 남은시간을 표시한다.
- 키보드만으로 업로드/취소/일괄취소/재시도 동작이 가능해야 한다 (KWCAG 2.2).
- 최소화 상태에서도 "완료 n / 전체 m"을 숫자로 표시한다.
- 진행률은 퍼센트 숫자 + 상태 텍스트를 함께 제공한다 (색상만 전달 금지).
- 스크린 리더용 `aria-live="polite"`는 유의미 분기점(시작, 50%, 완료, 실패)에서만 업데이트한다.
- 덮어쓰기/충돌 확인 모달은 Focus Trapping을 적용해 Tab이 모달 밖으로 이탈하지 않게 한다.

### 10-2) 업로드 API 정책

- 접수: `POST /api/v1/upload-jobs` (idempotency_key 필수)
- 상태조회: `GET /api/v1/upload-jobs/{job_id}`
- 파일별 취소: `POST /api/v1/upload-jobs/{job_id}/items/{item_id}/cancel`
- 일괄 취소: `POST /api/v1/upload-jobs/{job_id}/cancel-all`
- 재시도: `POST /api/v1/upload-jobs/{job_id}/items/{item_id}/retry`
- 중복 확인: `POST /api/v1/upload-jobs/check-duplicates` (파일명 + SHA-256)
- 덮어쓰기: `POST /api/v1/upload-jobs/{job_id}/items/{item_id}/overwrite` (target_file_id 필수)
- 청크: init / upload-part / complete / abort

### 10-3) 파일 검증 정책

- 허용 확장자 화이트리스트, MIME 화이트리스트, 파일 시그니처(매직넘버) 3중 검증을 적용한다.
- 첫 번째 청크(Header) 수신 시 python-magic으로 파일 시그니처를 즉시 검증한다 (조립 후 검사로는 늦다).
- 저장 파일명은 UUID 기반으로 생성하고, 원본 파일명은 DB 메타데이터로 분리 저장한다.
- 업로드 파일 저장 경로는 실행 권한을 제거하고 웹 실행 경로와 물리적으로 분리한다.

### 10-3-1) 파일 정체성(Identity) 정책 [결정: 채택]

- 파일의 고유 식별자는 **SHA-256 해시값**을 기본으로 한다.
- 같은 해시값을 가진 파일이 이미 존재할 경우:
  - 업로드 시도 시 "이미 동일한 내용의 파일이 존재합니다" 안내
  - 중복 저장하지 않으며, 기존 파일 ID를 반환한다.
  - 단, 사용자가 명시적으로 "덮어쓰기"를 선택한 경우는 예외로 한다.
- 파일명이 달라도 내용이 같으면 중복으로 감지한다.

### 10-4) 재업로드(이름 충돌) 정책 — 검색엔진 맞춤형

- 동일 파일명 충돌 시 선택 팝업을 제공한다:

```
⚠️ "동일한 이름의 파일(보고서.docx)이 이미 존재합니다. 어떻게 할까요?"

[ 최신 버전으로 업데이트 (기존 파일 덮어쓰기) ]
[ 건너뛰기 ]
```

- **'새 이름으로 저장(Keep Both)' 기능은 기본 제공하지 않는다.** CleverSearch는 통합 검색엔진이므로, 동일 내용의 과거/최신 버전이 동시에 검색 결과에 노출되면 사용자 혼란(검색 공해)을 유발한다.
- 다중 파일 업로드에서는 "이번 선택을 전체에 적용" 옵션을 제공한다.

### 10-4-1) 덮어쓰기 대상 명시화 정책 [결정: 채택]

- 같은 파일명이 여러 개 존재할 경우, 시스템은 자동으로 덮어쓸 수 없다.
- 사용자는 덮어쓸 대상 파일을 직접 선택해야 한다.
- 덮어쓰기 시:
  - 파일 ID는 유지하되, 버전을 1 증가시킨다.
  - 이전 버전은 히스토리로 보관하며, 일정 기간(기본 60일) 후 삭제 정책을 적용한다.
  - API 호출 시 `target_file_id`를 필수로 전달해야 한다.
- **동시성 제어(Optimistic Locking)**: 다수 관리자가 동시에 같은 문서를 덮어쓰는 충돌을 방지하기 위해 버전(version) 기반 제어를 적용한다.
- **무중단 교체**: 새 버전 문서의 색인이 완벽히 성공한 후에만 구 버전 문서를 삭제(또는 블라인드 처리)하여, 덮어쓰기 도중 검색 누락 공백기(Downtime)가 발생하지 않도록 한다.

### 10-4-2) '파일 교체' 버튼 vs 충돌 팝업 덮어쓰기 차이

| 구분          | 충돌 팝업의 '덮어쓰기'                     | 문서 목록의 '파일 교체' 버튼        |
| ------------- | ------------------------------------------ | ----------------------------------- |
| **진입 경로** | 업로드 시 같은 이름 감지                   | 기존 문서 옆 별도 버튼 클릭         |
| **대상 지정** | 파일명 기준 (자동 매칭)                    | 사용자가 직접 파일 선택             |
| **버전 관리** | 버전 1 증가                                | 버전 1 증가 (동일)                  |
| **사용 사례** | "아까 올린 파일을 최신으로 갱신"           | "특정 문서를 다른 파일로 완전 교체" |
| **주의사항**  | 파일명이 여러 개면 불가 (사용자 선택 필요) | 항상 가능                           |

구현상 두 기능은 내부적으로 같은 `overwrite` API를 호출하되, 파일명 충돌 시에만 팝업을 노출하는 UX 차이만 둔다.

### 10-5) 취소 후 정리 정책

- 취소 완료 기준은 UI 갱신이 아니라 OpenSearch+DB 정리 완료다.
- 취소 시 OpenSearch 문서 삭제와 DB 메타데이터 삭제를 같은 작업 단위로 실행한다.
- 부분 실패 시 `CANCELING` 상태를 유지하고 보상 재시도로 정합성을 회복한다.

### 10-6) 관리자 가시성 정책

- OpenSearch 내부 상세 필드는 문서 목록에 직접 노출하지 않는다.
- 관리자 전용 "검색 색인 진단" 메뉴를 분리해 인덱스 상태와 불일치 건수만 표시한다.
- 원문 JSON 조회는 최고권한에 한해 제한적으로 허용하고 감사 로그를 남긴다.

## 11. 신규 기능 아이디어 (Phase 3 이후 검토)

### 11-1) 임시 저장(Draft) 기능

- `draft=true` 파라미터로 업로드 접수하면 OpenSearch에 색인하지 않고 임시 스토리지에만 보관한다.
- 사용자가 "제출" 버튼을 누르면 실제 색인을 시작한다.
- 초안은 7일 후 자동 삭제한다.
- 사용 사례: 대용량 파일을 여러 개 올리다가 자리를 비워야 할 때, 최종 결정 전 문서를 임시로 보관할 때.

### 11-2) 스마트 중복 감지 (해시 기반)

- 파일명이 달라도 SHA-256 해시값이 같으면 중복 경고를 표시한다.

```
⚠️ "이 파일과 동일한 내용의 파일이 이미 존재합니다."

기존 파일: report_final.docx (2024-03-15)
[ 건너뛰기 ]  [ 그래도 업로드하기 ]  [ 기존 파일 위치 보기 ]
```

### 11-3) 업로드 히스토리 (타임라인 뷰)

- 관리자 페이지에 "내 업로드 이력" 탭을 추가한다.
- 표시 정보: 업로드 시간, 파일명, 상태(성공/실패/취소), 처리 시간.
- 실패 항목은 여기서 바로 "재시도" 가능.
- 대용량 업로드의 경우 청크별 전송 상태도 확인 가능 (운영자용).

### 11-4) 파일 교체 시 버전 비교 뷰어

- 파일 교체(덮어쓰기) 시 기존 버전과 새 버전의 차이를 미리 보여주는 기능.
- 텍스트 기반 파일(PDF, DOCX, HWP): 텍스트 추출 후 diff 뷰어로 추가/삭제/변경 하이라이트.
- 이미지 파일: 기존/새 이미지를 나란히 표시.
